# 创造·点击 — 一句话，一个模拟器

> 每个行业都有圈内人才懂的生存法则，我们让 AI 用 10 秒把它变成可以玩的模拟器。

---

## 为什么

**创作者的痛**：想做一个"程序员模拟器"或"投行生存游戏"？你得懂这个行业、会写代码、还得设计游戏数值。三道门槛，99% 的灵感死在了第一道。

**玩家的痛**：青椒模拟器火了（9 万 DAU），证明人们想玩"模拟自己行业"的游戏。但只有学术圈有一个，其他行业呢？没有人给房产中介、独立游戏开发者、急诊科医生做模拟器。

**我们的位置**：AI 已经足够懂每个行业的"圈内黑话"——大模型的训练语料里有无数行业老炮的经验帖、吐槽、攻略。我们提供一个固定的模块骨架，让 AI 往里面填充"只有圈内人才写得出"的内容。

**与创造·对话的关系**：
| | 创造·对话 (PRODUCT.md) | 创造·点击 (本文档) |
|--|----------------------|------------------|
| 交互模式 | 自由对话推进剧情 | 点击模块执行行动 |
| 内容生成 | AI 实时生成叙事文本 | AI 预生成模块内容 + 结果文本 |
| 时间模型 | 回合制（对话=回合） | 周期制（季度/学期/月） |
| 状态变化 | 对话中内嵌标记 | 行动直接修改数值 |
| 核心感受 | "我在演一部小说" | "我在经营一段人生" |

---

## 用户是谁

| 类型 | 一句话画像 | 来这里干嘛 |
|------|----------|----------|
| 行业人 | 在某个行业工作/学习的人 | 体验"如果重新选择"的平行人生 |
| 吃瓜群众 | 对某个行业好奇的外行 | 通过玩模拟器了解一个行业的真实生态 |
| 社交玩家 | 喜欢把结果截图发群里 | "我当了 3 年程序员，35 岁被优化了" |
| 内容创作者 | 短视频/图文博主 | "一句话生成 XX 模拟器"做选题 |

# 一、完整用户动线

> 一个用户从打开网页到分享结局，完整走一遍。

```
打开网页 → 输入一句话 → 等待生成 → 预览模拟器 → 进入模拟器 →
浏览模块 → 点击行动 → 看到结果 → 资源变化 → 时间推进 →
随机事件 → 做出抉择 → 阶段结算 → 继续下一周期 →
达成结局 → 结算画面 → 分享给朋友
```

小明打开链接，看到和"创造·对话"相同的创造终端——暗色全屏，中央输入框。但灵感提示不一样了：

# 二、核心设计原则

### 原则 1：圈内人视角 (Insider Voice)

这是区别于"通用模拟器"的核心竞争力。

**不要这样写**（通用 RPG 风格）：
> "工作：进行日常工作。经验+10，金钱+100。"

**要这样写**（圈内人视角）：
> "🔧 主导系统重构——把祖传代码翻新一遍，风险大但能出成果。"
> "📝 带新人做 onboarding——你的时间被碎片化了，但 mentor 这个标签晋升时有用。"
> "🏖 摸鱼划水——反正架构调整期间没人盯，先缓口气。"

AI 需要做到：
- 使用该行业的**真实术语**（P7/CR/述职/sponsor，不是"升级/战斗力"）
- 描述**真实的权衡**（"风险大但能出成果"，不是"获得经验"）
- 带有**行业老炮的语气**（略带自嘲、略带世故、略带无奈）
- 每个行动的风味文字让圈内人会心一笑，让圈外人窥见行业真相

### 原则 2：固定骨架 + 动态填充

**固定的**（模板层）：
- 模块数量和位置（5-7 个模块卡片）
- 每个模块的类型（主行动 / 社交 / 成长 / 理财 / ...）
- 资源栏的展示方式
- 事件流的 UI 结构
- 周期结算的流程

**动态的**（AI 生成）：
- 模块名称和 icon（从行业语境命名）
- 每个模块内的 3-4 个行动选项
- 每个行动的风味文字、成本、收益
- 随机事件的内容和选项
- 周期结算的叙事评语
- 结局条件和结局文本

### 原则 3：每周期刷新

每个周期开始时，AI 根据以下输入生成新的行动选项：
1. 当前资源状态（数值高低影响可用行动）
2. 已执行过的行动（不重复，或升级版出现）
3. 已发生的事件（事件改变可用路径）
4. 当前周期数（时间推进带来新机会和新压力）

这保证了**重玩性**——即使选同一个行业，不同的选择路径会导致完全不同的行动菜单。

# 三、功能用户故事

### C1 创造终端

**用户故事**：我想描述一个行业/身份，AI 帮我生成一个模拟器。

**复用创造·对话的 F1**，但灵感提示和按钮文案不同：
- 占位文字："用一句话，描述你想模拟的人生"
- 按钮文案："创造模拟器"
- 灵感提示偏向职业/行业/身份

---

### C2 模拟器生成加载

**用户故事**：AI 在生成模拟器，我想知道它在做什么。

**复用创造·对话的 F2**，阶段文字不同：
1. "正在分析行业背景…"
2. "正在构建模拟器骨架…"
3. "正在填充行动模块…"
4. "正在设计随机事件…"
5. "正在平衡数值系统…"
6. "正在调配视觉主题…"

---

### C3 模拟器预览

**用户故事**：生成完了，我想看看这个模拟器的设定是否合理。

**用户看到的**：
- 模拟器名称 + 类型标签
- 一段行业背景描述（圈内人口吻）
- 核心资源列表（icon + 名称 + 初始值）
- 行动模块预览（模块名 + 行动数量）
- 随机事件预览（3 条示例）
- 结局路线预览（4-6 种可能结局的名称）
- 主题配色预览
- 两个按钮："开始模拟" / "重新生成"

---

### C4 资源系统

**用户故事**：我需要清楚自己当前的状态——钱够不够？精力还剩多少？离晋升还差多远？

**触发**：游戏开始时初始化；每次行动后更新
**持续**：整个游戏过程

**用户看到的**（顶部状态栏）：
- 6-8 个核心资源，每个显示 icon + 名称缩写 + 数值
- 数值变化时有跳动动画 + 绿色(+)/红色(-) 提示
- 点击某个资源可展开详情（历史曲线、影响因素）

**资源类型**（由 AI 根据行业生成，以下为示例）：

| 行业 | 资源维度 |
|------|---------|
| 大厂程序员 | 现金 · 绩效 · 技术力 · 人脉 · 倦怠 · 职级 |
| 住院医师 | 工资 · 临床经验 · 科研点 · 医患关系 · 疲劳 · 职称 |
| 网文作者 | 稿费 · 字数存稿 · 读者热度 · 编辑关系 · 灵感 · 签约等级 |
| 餐厅主厨 | 资金 · 口碑 · 创意 · 团队士气 · 压力 · 星级 |

**特殊资源**：
- **行动力**：每周期固定点数（4-5 点），执行行动消耗，耗尽则周期结束
- **职级/段位**：阶梯型资源，达到门槛值跳级（P7→P8，住院→主治）

---

### C5 行动模块

**用户故事**：我想看看这个周期能做什么，点进去看详情，选一个执行。

**触发**：每个周期开始时，AI 生成/刷新模块内容
**持续**：整个游戏过程

**模块结构**（固定骨架）：

每个模拟器有 4-6 个行动模块，位置固定。AI 为每个模块命名并填充内容。

**每个模块内部**：
- 模块名 + icon
- 3-4 个行动选项
- 每个行动选项包含：
  - 行动名称 + icon
  - 风味描述（一句话，圈内人口吻）
  - 行动力成本（0-3 点）
  - 效果预览（哪些资源 ↑↓，部分可能标"不确定"）
  - [执行] 按钮

**行动选项的生成规则**：
- 每个选项必须有**权衡**（没有纯正面的选项）
- 效果可以是"不确定"（随机浮动，制造紧张感）
- 部分行动有前置条件（资源不够时灰显 + 提示缺什么）
- 已执行的行动本周期内不可重复

---

### C6 行动执行 → 结果叙事

**用户故事**：我点了一个行动，想看看发生了什么。结果不应该只是"数值+10"，而是一段让我身临其境的描述。

**触发**：用户点击 [执行] 按钮
**结束**：结果叙事展示完毕，资源更新

**用户看到的**：
- 弹出结果面板（或内联展开）
- AI 生成一段 2-4 句话的结果叙事（圈内人口吻）
- 资源变化列表（带动画）
- 如果触发了后续效果（解锁新行动、改变其他模块），也在此展示
- [关闭] 按钮

**结果叙事示例**：
> 你花了两个月把那坨 Java 8 时代的屎山用 Go 重写了一遍。CR 的时候老板看了眼 diff，说了句"还行"。但你知道真正的收益不在技术——而是晋升述职的时候，"主导核心系统重构"这六个字，值半个 P8。

**重要**：结果叙事是 AI 实时生成的（SSE 流式），不是预设文本。这保证了每次执行同一行动，叙事都不同。

---

### C7 随机事件

**用户故事**：突然弹出一个事件，迫使我做出选择。这种不可预测性让游戏有紧张感。

**触发**：每个周期内，按概率触发 0-2 个随机事件
**触发时机**：执行行动后、周期开始时、特定资源达到阈值时

**用户看到的**：
- 全屏半透明遮罩
- 居中事件卡片，从下方滑入
- 事件描述（2-3 句话，圈内人口吻）
- 2-3 个选项，每个选项有：
  - 选项文案（带情绪 emoji）
  - 效果预览（可能是"不确定"）
- 选择后展示结果叙事 + 资源变化

**事件类型**：
- **行业事件**：大环境变化（裁员潮、行业利好、政策变动）
- **人际事件**：同事/上级/客户带来的状况
- **个人事件**：身体/家庭/生活中的意外
- **机遇事件**：稍纵即逝的机会（跳槽 offer、投资机会）

**事件与行动的区别**：
| | 行动 | 随机事件 |
|--|------|---------|
| 触发方式 | 用户主动选择 | 系统随机触发 |
| 可预见性 | 效果基本已知 | 选项效果可能"不确定" |
| 频率 | 每周期 4-5 次 | 每周期 0-2 次 |
| 设计目的 | 给玩家掌控感 | 给玩家意外感 |

---

### C8 周期结算

**用户故事**：一个季度结束了，我想回顾这段时间做了什么、变化了什么。

**触发**：行动力耗尽时自动触发
**结束**：用户点击"进入下一周期"

**用户看到的**：
- 全屏结算面板
- 当前周期标题（如 "Q1 季度结算"）
- 本周期行动回顾列表（做了哪些事）
- 资源变化对比（每项资源的起始值→结束值）
- AI 生成的一句话评语（圈内人口吻）
- [进入下一周期] 按钮

**评语示例**：
> "不错的开局。但倦怠值在涨，别忘了你不是机器。"
> "这个季度你社交比干活多。小心，职场不是茶话会。"
> "闷头做技术是安全的，但安全不等于晋升。"

---

### C9 结局系统

**用户故事**：玩了一段时间，我达成了某个结局。我想看看完整的人生回顾。

**触发条件**（AI 预定义 4-6 种结局路线，每种有触发阈值）：
- 特定资源达到上限/下限
- 经过固定周期数（如 8 个季度）
- 特定事件链完成

**用户看到的**：
- 全屏结局画面
- 结局标题 + icon（🏆 / 💼 / 🔥 / 🏖 / 🚀）
- 结局叙事（AI 生成，3-5 段话，回顾整个模拟过程）
- 关键数据：总周期数、执行行动数、遭遇事件数
- 资源最终值一览
- 最高/最低资源高亮
- "重新来过" / "分享模拟器" 按钮

---

### C10 分享系统

**复用创造·对话的 F15**。

分享内容 = SimulatorConfig（不含游戏进度）。
朋友打开链接 → 直接预览模拟器设定 → 开始自己的模拟。

---

### C11 存档系统

**复用创造·对话的 F14**。

存档内容 = 当前周期 + 资源状态 + 已执行行动历史 + 已触发事件 + 模块当前内容。

# 四、数据模型

### SimulatorConfig（AI 一次性生成）

```
SimulatorConfig {
  title:          string        // "大厂生存指南"
  genre:          string        // "职场模拟"
  icon:           string        // "💼"
  description:    string        // 行业背景描述
  themeColors:    ThemeColors   // 主题配色
  narrativeStyle: string        // "程序员圈内老炮，略带自嘲"

  resources:      Resource[]    // 6-8 个核心资源定义
  modules:        Module[]      // 4-6 个行动模块骨架
  eventPool:      Event[]       // 20-30 个随机事件池
  endings:        Ending[]      // 4-6 种结局定义

  periodName:     string        // 周期名称（"季度"/"学期"/"月"）
  maxPeriods:     number        // 最大周期数
  actionPoints:   number        // 每周期行动力
}
```

### Resource（资源定义）

```
Resource {
  id:       string    // "cash"
  name:     string    // "现金"
  icon:     string    // "💰"
  unit:     string    // "万"
  initial:  number    // 120
  min:      number    // 0
  max:      number    // 9999
  color:    string    // "#4ade80"
  isLevel:  boolean   // false（true 表示阶梯型，如职级）
  levels:   string[]? // ["P6","P7","P8","P9"]（仅 isLevel=true）
}
```

### Module（行动模块骨架）

```
Module {
  id:       string    // "work"
  name:     string    // "🏢 工作"
  description: string // "日常开发、技术方案、跨团队协作"
  actions:  Action[]  // 每周期 AI 填充 3-4 个
}
```

### Action（行动选项 — AI 每周期生成）

```
Action {
  id:          string                    // "refactor-system"
  name:        string                    // "主导系统重构"
  icon:        string                    // "🔧"
  flavor:      string                    // "把祖传代码翻新……"
  cost:        number                    // 行动力消耗 2
  effects:     Record<string, number|'?'>  // { tech: +15, perf: +1, burnout: +20 }
  requires:    Record<string, number>?   // 前置条件 { tech: 60 }
  executed:    boolean                   // 本周期是否已执行
}
```

- `effects` 中值为 `'?'` 表示"不确定"，执行后随机决定

### Event（随机事件 — AI 预生成池）

```
Event {
  id:          string
  title:       string           // "老板突然约你 1v1"
  description: string           // 2-3 句话描述
  trigger:     EventTrigger     // 触发条件
  choices:     EventChoice[]    // 2-3 个选项
}

EventTrigger {
  type:   'random' | 'threshold' | 'period'
  chance: number?               // random 时的概率
  condition: Record<string, number>?  // threshold 时的资源条件
  period:  number?              // period 时的周期数
}

EventChoice {
  text:    string               // "😰 紧张地问'什么事'"
  effects: Record<string, number|'?'>
}
```

### Ending（结局定义）

```
Ending {
  id:         string
  title:      string        // "晋升 P8"
  icon:       string        // "🏆"
  condition:  EndingCondition
  priority:   number        // 多个结局同时满足时取优先级最高的
}

EndingCondition {
  type:      'resource_gte' | 'resource_lte' | 'period_max' | 'event_chain'
  resource:  string?
  value:     number?
  events:    string[]?      // 事件链 ID
}
```

### GameState（运行时状态）

```
GameState {
  simulatorConfig: SimulatorConfig
  currentPeriod:   number
  actionPoints:    number          // 剩余行动力
  resources:       Record<string, number>
  modules:         Module[]        // 含当前周期的 actions
  executedActions:  string[]       // 本周期已执行
  allActions:       string[]       // 全局已执行（避免重复生成）
  triggeredEvents:  string[]       // 已触发的事件 ID
  eventLog:         EventLogEntry[]  // 事件流时间线
  actionLog:        ActionLogEntry[] // 行动历史
  ended:            boolean
  endingId:         string?
}
```

# 五、AI 调用策略

与创造·对话不同，点击驱动的模拟器需要两种 AI 调用模式：

### 模式 1：一次性生成（创造阶段）

用户输入一句话 → AI 生成完整的 `SimulatorConfig`。

**Prompt 要求**：
- 以该行业/身份的"圈内人"视角构建所有内容
- 资源维度要反映行业真实的关注点（不是通用的 HP/MP）
- 行动模块要覆盖该身份的主要生活面向
- 随机事件要真实可信（来自行业真实场景）
- 结局路线要反映行业真实的可能走向

**调用模型**：高能力模型（Gemini Flash / Claude），一次调用生成完整 JSON。

### 模式 2：周期刷新（游戏中）

每个新周期开始时 → AI 根据当前状态生成新的行动选项。

**输入**：
```
当前周期: Q3
当前资源: { cash: 95w, perf: A-, tech: 90, network: 60, burnout: 65, level: P7 }
已做过的行动: [系统重构, 技术峰会, 带新人, ...]
近期事件: [被老板 1v1, 竞对挖人, ...]
```

**输出**：每个模块 3-4 个新行动选项（含风味文字和效果）。

**调用模型**：快速模型（Gemini Flash），低延迟优先。

### 模式 3：结果叙事（行动执行时）

用户点击 [执行] → AI 生成一段结果叙事。

**输入**：行动详情 + 当前状态 + 近期上下文
**输出**：2-4 句话的圈内人视角叙事
**调用方式**：SSE 流式输出，体验流畅

# 六、与青椒模拟器的差异化

| 维度 | 青椒模拟器 | 创造·点击 |
|------|----------|----------|
| 行业覆盖 | 仅学术圈 | 任何行业（AI 生成） |
| 内容来源 | 人工编写 | AI 从圈内人视角生成 |
| 创建方式 | 固定一个 | 一句话创造任意模拟器 |
| 分享机制 | 无 | URL 压缩分享 |
| 重玩性 | 同一套内容 | 每周期 AI 刷新行动 |
| 结果叙事 | 简短提示 | 流式生成圈内人叙事 |
| 技术栈 | 不明 | 纯前端，零后端依赖 |

**我们的核心优势**：AI 让"一个模拟器"变成"无限个模拟器"。青椒做学术圈，我们做所有圈。
